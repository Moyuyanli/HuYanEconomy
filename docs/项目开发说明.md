## 1. 开发环境搭建

### 1.1 基础环境

| 工具 | 版本要求 | 说明 |
|------|---------|------|
| JDK | 11+ | JVM Target 为 11，不要使用 11 以上独有的 API |
| Kotlin | 1.9.20 | 由 Gradle 自动管理 |
| Gradle | 8.x | 使用项目自带的 Gradle Wrapper |
| IDE | IntelliJ IDEA / VS Code | 推荐 IntelliJ IDEA，支持 KSP 和 mirai-console 调试 |

### 1.2 克隆项目

```bash
git clone https://github.com/Moyuyanli/HuYanEconomy.git
cd HuYanEconomy
```

### 1.3 IDE 配置

**IntelliJ IDEA：**
1. 打开项目根目录，等待 Gradle 同步完成
2. 安装插件：`Kotlin`、`Gradle`（通常自带）
3. 确认 JDK 版本设置为 11

**VS Code：**
1. 安装扩展：`Kotlin`、`Gradle for Java`
2. 确保 `JAVA_HOME` 指向 JDK 11+

### 1.4 关键依赖说明

| 依赖 | 作用 | 备注 |
|------|------|------|
| `mirai-console` 2.16.0 | QQ 机器人框架 | 提供插件生命周期、事件系统 |
| `HuYanAuthorize` | 权限系统 | KSP 注解处理，自动扫描 `action/` 包 |
| `hibernate-plus` | ORM 封装 | 提供 `HibernateFactory` 工具类 |
| `mirai-hibernate-plugin` | Hibernate 运行时 | **不要打入插件 jar**，由运行时提供 |
| `hutool-all` | 工具库 | 集合、日期、随机数等工具 |
| `poi-ooxml` | 图片生成 | Apache POI + Java AWT 生成签到图等 |
| `overflow-core-api` | OneBot 兼容层 | 可选，用于 OneBot 协议支持 |

> ⚠️ **重要**：`mirai-economy-core`、`HuYanAuthorize`、`overflow-core-api` 使用 `compileOnly`，运行时由 mirai-console 环境提供。

## 2. 构建与测试

### 2.1 构建命令

```bash
# 标准构建（输出到 build/mirai/ 目录）
./gradlew.bat buildPlugin --console=plain

# 构建并查看完整堆栈（调试构建失败时使用）
./gradlew.bat buildPlugin --console=plain --stacktrace

# 构建并保存日志到文件
./gradlew.bat buildPlugin --console=plain --stacktrace 2>&1 | Tee-Object -FilePath .\buildplugin.log

# 六模块测试
./gradlew.bat :economy-common:test :economy-data:test :economy-image:test :economy-core:test :economy-game:test :economy-main:test --console=plain

# 只验证入口相关编译
./gradlew.bat :economy-game:compileKotlin :economy-main:compileKotlin --console=plain
```

构建产物位于 `build/mirai/` 目录，生成的 jar 文件可直接放入 mirai-console 的 `plugins/` 目录。

### 2.2 构建流程说明

1. **模块编译**：先编译 `economy-common/data/image/core/game`，再编译 `economy-main`
2. **KSP 注解处理**：在 `economy-main` 中扫描 `cn.chahuyun.economy.action` 包，生成 `AuthorizeRegistrar`（位于 `economy-main/build/generated/ksp/`）
3. **BuildConfig 生成**：在 `economy-common` 生成 `EconomyBuildConstants`，包含版本号和构建时间
4. **Mirai 插件打包**：由 `economy-main:buildPlugin` 生成插件 jar
5. **根任务同步**：根 `buildPlugin` 将 `economy-main/build/mirai/` 下的插件产物同步到 `build/mirai/`

> ⚠️ `economy-main/build/generated/ksp/` 下的生成代码**不要手动修改**，每次构建会自动重新生成。

### 2.3 本地测试运行

1. 将构建好的 jar 复制到 `debug-sandbox/plugins/` 目录
2. 配置 `debug-sandbox/config/cn.chahuyun.HuYanEconomy/` 下的配置文件
3. 配置 `debug-sandbox/bots/` 下的 bot QQ 号目录
4. 启动 mirai-console 测试

## 3. 项目架构

### 3.1 模块架构

2.0.0 之后项目采用 Gradle 多模块结构，主产物仍由 `economy-main` 打包为 Mirai 插件 jar。

```
economy-main    → Mirai 插件入口、Action、命令、事件、权限注册、最终打包
economy-common  → 通用基础能力、常量、时间/金额/文本工具
economy-data    → 实体、DTO、Converter、Repository、EntityProxy、数据版本与缓存基础设施
economy-image   → 图片渲染、底图、字体、画布工具、Renderer
economy-core    → 签到、银行、背包、道具、称号、红包、私人银行等核心经济流程
economy-game    → 钓鱼、抢劫、抽奖、农场等玩法流程
```

依赖方向：

```
economy-main  -> economy-core, economy-game, economy-image, economy-data, economy-common
economy-game  -> economy-core, economy-data, economy-image, economy-common
economy-core  -> economy-data, economy-image, economy-common
economy-data  -> economy-common
economy-image -> economy-common
economy-common -> 无项目内模块依赖
```

当前阶段约束：

- 不创建 `economy-api` / `economy-proxy` 模块。
- `proxy` 只表示数据代理，归入 `economy-data`。
- `economy-core -> economy-image` 是当前迁移期保留的实际依赖，不再扩大到 image 之外。
- `economy-core` / `economy-game` 短期允许 Mirai 类型和发消息逻辑。
- 新增数据读写优先走 `economy-data` 的 repository/proxy，不在 core/game 中新增 Hibernate 直连。
- 不动 `EconomyUtil` 账户资金语义。

### 3.2 逻辑分层

```
action/          → 控制层：指令入口，由 AuthorizeServer 扫描注册事件
  ↓
usecase/         → 业务层：核心业务逻辑
  ↓
manager/         → 模块管理器：模块级 API 与生命周期管理
  ↓
repository/      → 数据访问层
  ↓
entity/          → 数据实体（Hibernate @Entity）
```

**数据流向**：用户消息 → Action（指令匹配）→ Usecase（业务处理）→ Manager（模块 API）→ Repository（数据持久化）→ Entity（数据模型）

### 3.4 插件生命周期

插件主类 `HuYanEconomy`（`object` 单例，继承 `KotlinPlugin`）的生命周期：

```
onLoad()    → 设置插件状态标志
    ↓
onEnable()  → 加载配置 → 注册命令 → 初始化 Manager → 注册事件监听
    ↓
运行中...    → 处理用户消息和事件
    ↓
onDisable() → 关闭 Manager、调度器，清理资源
```

`onEnable()` 中的初始化顺序：

1. 加载配置文件（`EconomyConfig`, `FishingMsgConfig` 等）
2. 注册 mirai-console 命令（`EconomyCommand`）
3. 初始化工具类（`MessageUtil`, `PluginManager`）
4. 注册权限码（`PermCodeManager.init()`）
5. 初始化数据库（`HibernateUtil.init()`）
6. 初始化各模块 Manager（`LotteryManager`, `FishManager`, `BankManager` 等）
7. 注册 Action 事件扫描（`AuthorizeServer.registerEvents()`）
8. 注册自定义事件监听（`SignEvent`, `FishStartEvent` 等）

## 4. 如何添加新功能

以添加一个「每日答题」功能为例，演示完整的开发流程。

> 2.0.0 之后新增功能优先按多模块归属落位：数据结构和仓储进入 `economy-data`，核心流程进入 `economy-core`，玩法流程进入 `economy-game`，Mirai 指令入口进入 `economy-main`。不要在 core/game 中新增 `HibernateFactory` 直连。

### 4.1 第一步：创建实体类（entity）

在 `economy-data/src/main/kotlin/cn/chahuyun/economy/entity/` 下创建 JPA 实体：

```kotlin
package cn.chahuyun.economy.entity

import jakarta.persistence.Entity
import jakarta.persistence.Id
import jakarta.persistence.Table
import java.util.Date

/**
 * 用户答题记录
 */
@Entity(name = "UserQuiz")
@Table(name = "UserQuiz")
class UserQuiz(
    @Id
    var id: Long? = null,

    /** 用户 QQ 号 */
    var qq: Long = 0,

    /** 今日是否已答题 */
    var answered: Boolean = false,

    /** 最后答题时间 */
    var lastTime: Date? = null,

    /** 累计答对次数 */
    var correctCount: Int = 0
)
```

**注意事项：**
- 使用 `@Entity` 和 `@Table` 注解
- 必须有 `@Id` 主键字段
- 包名保持在 `cn.chahuyun.economy.entity` 下，便于 Hibernate 实体注册和历史兼容
- 如果字段名是数据库关键字，使用 `@Column(name = "\`force\`")` 反引号转义

### 4.2 第二步：创建数据访问层（repository）

在 `economy-data/src/main/kotlin/cn/chahuyun/economy/data/repository/` 下创建 Repository（如需要复杂查询）：

```kotlin
package cn.chahuyun.economy.data.repository

import cn.chahuyun.economy.entity.UserQuiz

/**
 * 答题数据持久化层
 */
object QuizRepository {

    @JvmStatic
    fun findByQq(qq: Long): UserQuiz? =
        HibernateDataStore.selectOne(UserQuiz::class.java, "qq", qq)

    @JvmStatic
    fun save(quiz: UserQuiz): UserQuiz =
        HibernateDataStore.merge(quiz)
}
```

**常用 data 仓储 API：**
- `HibernateDataStore.merge(entity)` — 保存或更新实体
- `HibernateDataStore.selectOne(Class, field, value)` — 按单字段查询
- `HibernateDataStore.selectList(Class, field, value)` — 按单字段查询列表
- `HibernateDataStore.selectList(Class)` — 查询全部
- 自定义事务应集中封装在 `economy-data` repository 内部，不要散落到 core/game。

### 4.3 第三步：创建模块管理器（manager）

核心经济功能在 `economy-core/src/main/kotlin/cn/chahuyun/economy/manager/` 下创建 Manager；钓鱼、抢劫、抽奖、农场等玩法功能优先放入 `economy-game`。

```kotlin
package cn.chahuyun.economy.manager

import cn.chahuyun.economy.utils.Log

/**
 * 答题模块管理器
 */
object QuizManager {

    @JvmStatic
    fun init() {
        // 初始化逻辑（如加载题库等）
        Log.info("答题模块初始化完成")
    }

    @JvmStatic
    fun shutdown() {
        // 清理资源（如有定时任务等）
    }
}
```

**Manager 职责：**
- 提供模块级 API 供 Usecase 调用
- 管理模块的初始化和销毁生命周期
- 如需要启动注册，在 `economy-main` 的 `HuYanEconomy.onEnable()` 中调用 `QuizManager.init()`
- 如有清理逻辑，在 `onDisable()` 中调用 `QuizManager.shutdown()`

### 4.4 第四步：创建业务逻辑层（usecase）

核心经济功能在 `economy-core/src/main/kotlin/cn/chahuyun/economy/usecase/` 下创建 Usecase；玩法功能放入 `economy-game` 的 `usecase/`。

```kotlin
package cn.chahuyun.economy.usecase

import cn.chahuyun.economy.entity.UserQuiz
import cn.chahuyun.economy.manager.QuizManager
import cn.chahuyun.economy.data.repository.QuizRepository
import cn.chahuyun.economy.utils.EconomyUtil
import cn.chahuyun.economy.utils.Log
import cn.chahuyun.economy.utils.MessageUtil
import net.mamoe.mirai.event.events.GroupMessageEvent
import net.mamoe.mirai.message.data.PlainText

/**
 * 答题业务逻辑
 */
object QuizUsecase {

    /**
     * 开始答题
     */
    suspend fun startQuiz(event: GroupMessageEvent) {
        val user = event.sender
        val subject = event.subject
        val message = event.message
        val reply = MessageUtil.quoteReply(message)

        // 查询或创建用户答题记录
        var quiz = QuizRepository.findByQq(user.id)
        if (quiz == null) {
            quiz = UserQuiz(qq = user.id)
        }

        // 检查今日是否已答题
        if (quiz.answered) {
            subject.sendMessage(reply.append(PlainText("你今天已经答过题了哦！")).build())
            return
        }

        // 业务逻辑...
        quiz.answered = true
        QuizRepository.save(quiz)

        // 发放奖励
        EconomyUtil.plusMoneyToUser(user, 100.0)

        subject.sendMessage(reply.append(PlainText("回答正确！获得 100 金币！")).build())
    }
}
```

**Usecase 职责：**
- 编排业务流程，协调 Manager 和 Repository
- 处理消息的发送和回复
- 使用 `object` 单例模式

### 4.5 第五步：创建指令入口（action）

在 `economy-main/src/main/kotlin/cn/chahuyun/economy/action/` 下创建 Action 类：

```kotlin
package cn.chahuyun.economy.action

import cn.chahuyun.authorize.EventComponent
import cn.chahuyun.authorize.MessageAuthorize
import cn.chahuyun.authorize.constant.AuthPerm
import cn.chahuyun.economy.usecase.QuizUsecase
import net.mamoe.mirai.event.events.GroupMessageEvent

/**
 * 答题指令入口
 */
@EventComponent
class QuizAction {

    /**
     * 每日答题
     */
    @MessageAuthorize(text = ["答题", "每日答题", "quiz"])
    suspend fun quiz(event: GroupMessageEvent) {
        QuizUsecase.startQuiz(event)
    }

    /**
     * 管理员重置答题（仅群主/管理员可用）
     */
    @MessageAuthorize(
        text = ["重置答题"],
        userPermissions = [AuthPerm.OWNER, AuthPerm.ADMIN]
    )
    suspend fun resetQuiz(event: GroupMessageEvent) {
        QuizUsecase.resetQuiz(event)
    }
}
```

**Action 注解说明：**
- `@EventComponent` — 标记此类为事件组件，AuthorizeServer 会自动扫描
- `@MessageAuthorize` — 标记消息处理方法
  - `text` — 触发关键词列表（匹配任意一个即触发）
  - `userPermissions` — 用户需要的权限（AND 关系）
  - `blackPermissions` — 黑名单权限（拥有则拒绝）
  - `groupPermissions` — 群需要的权限

**注意事项：**
- Action 类的包必须在 `cn.chahuyun.economy.action` 下
- 方法参数必须是具体的事件类型（如 `GroupMessageEvent`）
- 方法使用 `suspend` 修饰（支持协程）
- Action 只负责接收消息和调用 Usecase，不包含业务逻辑

### 4.6 第六步：在主类中初始化

如新功能需要启动初始化，在 `economy-main` 的 `HuYanEconomy.onEnable()` 中添加 Manager 初始化：

```kotlin
override fun onEnable() {
    // ...existing code...

    // 功能加载
    // ...existing code...
    QuizManager.init()  // ← 添加这一行

    // ...existing code...
}
```

如果 Manager 有清理逻辑，在 `onDisable()` 中添加：

```kotlin
override fun onDisable() {
    // ...existing code...
    QuizManager.shutdown()  // ← 添加这一行
    // ...existing code...
}
```

> **无需手动注册 Action**：`AuthorizeServer.registerEvents()` 会自动扫描 `cn.chahuyun.economy.action` 包下所有带 `@EventComponent` 注解的类。

### 4.7 第七步：注册权限码（如需要）

如果功能需要权限控制，在 `economy-common` 的 `constant/EconPerm.kt` 中添加常量：

```kotlin
object EconPerm {
    // ...existing code...
    const val QUIZ_PERM: String = "quiz"

    object GROUP {
        // ...existing code...
        const val QUIZ_PERM_GROUP: String = "答题组"
    }
}
```

然后在 `economy-main` 的 `plugin/PermCodeManager.init()` 中注册：

```kotlin
AuthorizeServer.registerPermissions(
    plugin,
    // ...existing code...
    Perm(EconPerm.QUIZ_PERM, "壶言经济的答题权限"),
)
```

### 4.8 完整流程总结

```
1. economy-data   → 创建实体、DTO、Converter、Repository / EntityProxy
2. economy-core   → 创建核心 Manager / Usecase / Service
3. economy-game   → 创建玩法 Manager / Usecase（仅玩法功能需要）
4. economy-image  → 创建图片渲染器和渲染输入模型（仅图片输出需要）
5. economy-main   → 创建 Action、命令、事件入口和启动注册
6. economy-common → 添加平台弱相关常量或基础工具（确实通用时才放）
7. 权限码         → 在 EconPerm 和 PermCodeManager 中注册（如需要）
```

模块落位判断：

| 类型 | 推荐模块 | 说明 |
|------|---------|------|
| 新指令入口、Console 命令、Mirai 事件监听 | `economy-main` | 入口层保持薄，只做参数解析和分发 |
| 签到、银行、背包、称号、红包、私人银行 | `economy-core` | 核心经济流程与通用经济能力 |
| 钓鱼、抢劫、抽奖、农场 | `economy-game` | 玩法流程、玩法视图、玩法服务 |
| 实体、DTO、Converter、Repository、Proxy、缓存 | `economy-data` | 数据访问和版本兼容集中管理 |
| 图片渲染器、绘制模型、画布工具 | `economy-image` | 不做 Mirai 上传和消息发送 |
| 权限码、通用 Result、时间/金额/文本工具 | `economy-common` | 不依赖 Mirai、Hibernate 或业务模块 |

开发完成后的最小验证建议：

```bash
# 改了单个模块时先跑对应模块测试
./gradlew.bat :economy-game:test --console=plain

# 改了入口、注解、构建或跨模块契约时跑六模块测试
./gradlew.bat :economy-common:test :economy-data:test :economy-image:test :economy-core:test :economy-game:test :economy-main:test --console=plain

# 准备交付插件时跑完整打包
./gradlew.bat buildPlugin --console=plain
```

### 4.9 新版本道具开发规则

从当前版本开始，功能性道具统一采用“道具状态 + 效果处理器 + 注册表”的设计。

核心原则：

- 道具模型只保存状态，不直接操作其他模块。
- 功能效果写在 `prop/effect/` 下的 `PropEffectHandler` 中。
- `FunctionProps.use()` 只负责把使用请求委托给 `PropEffectRegistry`。
- 新增功能性道具时，不再往 `FunctionProps.use()` 里追加 `when(code)` 分支。
- 需要和签到、抢劫、钓鱼、禁言、用户属性等模块交互时，由 handler 或后续抽象端口承接，不把业务逻辑塞进道具数据类。

#### 4.9.1 文件职责

| 文件/目录 | 职责 |
|---------|------|
| `model/props/FunctionProps.kt` | 功能道具的数据状态与展示文本，例如 `electricity`、`muteTime`、`num` |
| `model/props/PropsCard.kt` | 卡片道具的数据状态与展示文本，例如 `status`、`expiredTime` |
| `prop/effect/PropEffect.kt` | 道具效果接口与注册表 |
| `prop/effect/*Effects.kt` | 具体道具效果实现 |
| `plugin/PluginPropsManager.kt` | 注册道具类型、道具模板、道具效果处理器 |
| `prop/PropsManager.kt` | 统一处理道具使用、消耗、更新、删除 |

#### 4.9.2 新增功能性道具流程

1. 在 `FunctionProps` 的 `companion object` 中新增道具 code。

```kotlin
companion object {
    const val MY_PROP = "my-prop"
}
```

2. 在 `PluginPropsManager.initFunctionProps()` 中添加道具模板。

```kotlin
FunctionProps(PropsKind.functionProp, FunctionProps.MY_PROP, "我的道具").apply {
    description = "道具说明"
    cost = 100
    canBuy = true
    isConsumption = true
    unit = "个"
}
```

3. 在 `prop/effect/` 下新增效果处理器。

```kotlin
object MyPropEffectHandler : PropEffectHandler {
    override val codes: Set<String> = setOf(FunctionProps.MY_PROP)

    override suspend fun use(prop: BaseProp, event: UseEvent): UseResult {
        val functionProp = prop as? FunctionProps
            ?: return UseResult.fail("该道具数据异常!")

        // 执行业务效果
        return UseResult.success("${functionProp.name} 使用成功")
    }
}
```

4. 在 `PluginPropsManager.init()` 中注册效果处理器。

```kotlin
PropEffectRegistry.registerAll(
    MyPropEffectHandler
)
```

#### 4.9.3 可堆叠与消耗规则

- 可堆叠道具必须实现 `Stackable` 且 `isStack = true`。
- 不可堆叠道具即使实现了 `Stackable`，也必须显式设置 `isStack = false`。
- 一次性消耗道具设置 `isConsumption = true`，使用成功后由 `PropsManager` 统一扣数量或删除背包记录。
- 耐久型道具设置 `isConsumption = false`，在 handler 内修改自身状态，例如电棍扣 `electricity`，再由 `PropsManager` 统一保存。
- 商店模板必须视为只读模板。发放道具时必须复制模板，不允许直接把商店模板当作用户实例保存。

#### 4.9.4 和其他模块关联的规则

- 道具模型不得直接调用 `FactorManager`、`UserCoreManager`、`BackpackManager`、`Group.mute()` 等模块能力。
- 主动使用型道具由 handler 承接效果，例如红牛、禁言卡、改名卡。
- 被动触发型道具由对应业务模块在自己的流程中检测或触发，例如电棍在抢劫流程中触发、月卡在签到流程中生效。
- 需要用户二次输入的道具，优先在 Usecase 层解析参数，再传入使用上下文；handler 只承接明确的业务效果。
- 后续如果某个道具需要访问复杂模块能力，优先抽象端口接口，例如 `FactorPort`、`ModerationPort`，再由 handler 依赖端口，而不是直接依赖具体 Manager。

#### 4.9.5 禁止做法

```kotlin
// 不要这样：继续在 FunctionProps.use() 中追加功能分支
override suspend fun use(event: UseEvent): UseResult {
    return when (code) {
        MY_PROP -> {
            // 业务逻辑
        }
        else -> UseResult.fail("该道具无法直接使用!")
    }
}
```

正确做法是新增 `PropEffectHandler`，并在 `PropEffectRegistry` 注册。

## 7. 数据库操作

### 7.1 数据库配置

在 `config/cn.chahuyun.HuYanEconomy/config.yml` 中配置：

```yaml
dataType: H2        # 可选: H2, SQLITE, MYSQL
mysqlUrl: 'localhost:3306/test'
mysqlUser: 'root'
mysqlPassword: '123456'
```

| 数据库 | 适用场景 | 数据文件位置 |
|--------|---------|-------------|
| H2 | 开发/测试（默认） | `data/HuYanEconomy.h2.mv.db` |
| SQLite | 小型部署 | `data/HuYanEconomy` |
| MySQL | 生产环境 | 远程数据库 |

### 7.2 数据仓储常用操作

2.0.0 后，业务模块不应新增 `HibernateFactory` 直连。数据库访问应集中在 `economy-data` 的 repository / proxy 内部，业务侧通过 data 提供的入口访问。

```kotlin
import cn.chahuyun.economy.data.repository.HibernateDataStore

// 保存/更新实体
HibernateDataStore.merge(entity)

// 按字段查询单条
val user = HibernateDataStore.selectOne(UserInfo::class.java, "qq", 12345L)

// 按字段查询列表
val list = HibernateDataStore.selectList(FishInfo::class.java, "status", true)

// 查询全部
val all = HibernateDataStore.selectList(UserInfo::class.java)
```

复杂事务也应封装在 `economy-data` repository 内部，再由 core/game 调用，不要把事务细节散落到业务流程中。

历史代码中仍可能存在 `HibernateFactory` 相关封装；迁移时优先将它们收敛到 `economy-data`。

### 7.3 实体类映射

- 实体类必须在 `cn.chahuyun.economy.entity` 包下（HibernateUtil 扫描此包）
- 新增实体类后**自动建表**，无需手动创建
- 字段类型映射：Kotlin `String` → `VARCHAR`，`Long` → `BIGINT`，`Boolean` → `BOOLEAN`，`Date` → `TIMESTAMP`

### 7.4 版本升级数据库修复

当实体类结构变更（新增字段、修改类型等）时：

1. 更新实体类
2. 运行 `hye repair` 查看范围，再按数据版本执行对应修复
3. 修复逻辑在 `repair/RepairManager` 中实现

## 8. 权限系统使用

### 8.1 权限系统概述

项目使用 [HuYanAuthorize](https://github.com/Moyuyanli/HuYanAuthorize) 权限系统，基于 KSP 注解处理，自动扫描 `action/` 包下的 `@EventComponent` 类。

### 8.2 Action 权限控制

```kotlin
@EventComponent
class MyAction {

    // 无权限要求 - 所有人可用
    @MessageAuthorize(text = ["签到"])
    suspend fun sign(event: GroupMessageEvent) { /* ... */ }

    // 需要特定用户权限
    @MessageAuthorize(
        text = ["关闭签到"],
        userPermissions = [AuthPerm.OWNER, AuthPerm.ADMIN]
    )
    suspend fun offSign(event: GroupMessageEvent) { /* ... */ }

    // 黑名单权限 - 拥有此权限的用户无法使用
    @MessageAuthorize(
        text = ["签到"],
        blackPermissions = [EconPerm.SIGN_BLACK_PERM]
    )
    suspend fun signWithBlacklist(event: GroupMessageEvent) { /* ... */ }

    // 自定义权限
    @MessageAuthorize(
        text = ["钓鱼"],
        userPermissions = [EconPerm.FISH_PERM]
    )
    suspend fun fish(event: GroupMessageEvent) { /* ... */ }
}
```

### 8.3 内置权限

在 `AuthPerm`（HuYanAuthorize 提供）中：

| 权限 | 说明 |
|------|------|
| `AuthPerm.OWNER` | 群主 |
| `AuthPerm.ADMIN` | 管理员 |

### 8.4 自定义权限

**步骤 1：定义权限常量**

在 `constant/EconPerm.kt` 中添加：

```kotlin
object EconPerm {
    const val MY_PERM: String = "my-permission"

    object GROUP {
        const val MY_PERM_GROUP: String = "我的权限组"
    }
}
```

**步骤 2：注册权限**

在 `plugin/PermCodeManager.init()` 中添加：

```kotlin
AuthorizeServer.registerPermissions(
    plugin,
    // ...existing code...
    Perm(EconPerm.MY_PERM, "权限描述"),
)
```

**步骤 3：创建权限组（可选）**

```kotlin
var group = util.selectPermGroupOneByName(EconPerm.GROUP.MY_PERM_GROUP)
if (group == null) {
    group = util.takePermGroupByName(EconPerm.GROUP.MY_PERM_GROUP)
    val perm = util.takePerm(EconPerm.MY_PERM)
    util.addPermToPermGroupByPermGroup(perm, group)
}
```

### 8.5 代码中检查权限

```kotlin
import cn.chahuyun.authorize.utils.PermUtil
import cn.chahuyun.authorize.utils.UserUtil

// 检查用户是否有某权限
val hasPerm = PermUtil.hasPerm(userId, EconPerm.FISH_PERM)
```

## 9. 常见开发问题

### 9.1 ClassLoader 冲突（LinkageError）

**现象**：运行时出现 `LinkageError: loader constraint violation`，涉及 `log4j` 或 `AbstractLogger`。

**原因**：mirai-console 的 app classloader 已包含 log4j-api，插件的 private classloader 不应再包含。

**解决**：
- `build.gradle.kts` 中已全局排除 log4j：`configurations.configureEach { exclude(group = "org.apache.logging.log4j") }`
- 新增依赖时，如果传递依赖包含 log4j，需要手动排除：
  ```kotlin
  implementation("some:library:1.0") {
      exclude(group = "org.apache.logging.log4j")
  }
  ```

### 9.2 Hibernate 类冲突（ServiceConfigurationError）

**现象**：出现 `ServiceConfigurationError: "xxx not a subtype"`。

**原因**：Hibernate/HikariCP 被打入插件 jar，与 mirai-hibernate-plugin 提供的运行时版本冲突。

**解决**：
- **不要**将 Hibernate 核心依赖设为 `implementation`，运行期由 mirai-hibernate-plugin 提供
- 使用 `compileOnly` 引用 Hibernate API

### 9.3 KSP 生成代码被修改

**现象**：构建时报错或权限注册不生效。

**原因**：手动修改了 `economy-main/build/generated/ksp/` 下的生成代码。

**解决**：KSP 生成代码每次构建会自动重新生成，**不要手动修改**。如需调整，修改源码中的注解。

### 9.4 数据库表不存在

**现象**：查询时报错 `Table not found`。

**原因**：新增实体类后未建表。

**解决**：
- H2/SQLite：Hibernate 通常自动建表
- 如未自动建表，先检查 Hibernate DDL 配置；`hye repair` 仅处理已登记的数据修复动作

### 9.5 Action 不生效

**现象**：发送关键词无响应。

**排查步骤**：
1. 确认类在 `cn.chahuyun.economy.action` 包下
2. 确认类有 `@EventComponent` 注解
3. 确认方法有 `@MessageAuthorize` 注解
4. 确认 `text` 中的关键词与发送内容匹配
5. 确认没有被 `blackPermissions` 拦截
6. 查看日志是否有错误信息

### 9.6 配置文件不生效

**现象**：修改配置后不生效。

**排查步骤**：
1. 确认配置类继承 `AutoSavePluginConfig`
2. 确认在 `onEnable()` 中调用了 `reload()`
3. 确认使用 `by value()` 委托属性
4. 重启插件使配置生效

### 9.7 日志输出不显示

**现象**：调用 `Log.info()` 但控制台无输出。

**排查**：
- 确认使用 `cn.chahuyun.economy.utils.Log`，不是 `java.util.logging.Logger`
- 确认插件已正常加载（`HuYanEconomy.logger` 已初始化）
- 检查日志级别设置

### 9.8 构建失败排查

```bash
# 查看完整堆栈
./gradlew.bat buildPlugin --console=plain --stacktrace

# 保存日志到文件
./gradlew.bat buildPlugin --console=plain --stacktrace 2>&1 | Tee-Object -FilePath .\buildplugin.log
```

常见构建失败原因：
- JDK 版本不是 11
- 依赖下载失败（网络问题）
- Kotlin 编译错误（语法问题）
- KSP 处理失败（注解使用错误）
