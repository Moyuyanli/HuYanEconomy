## 2. 当前事实基线

### 2.1 模块结构

`settings.gradle.kts` 当前只包含六个正式模块：

- `economy-common`
- `economy-data`
- `economy-image`
- `economy-core`
- `economy-game`
- `economy-main`

当前明确不创建：

- `economy-api`
- 平台入口意义上的 `economy-proxy`

当前约定：

- `proxy` 只表示数据代理，归入 `economy-data`。
- 主产物仍是 Mirai 插件 jar，由 `economy-main` 负责最终打包。
- `core/game` 短期允许 Mirai 类型与 Mirai 发消息存在。
- 不强制抽平台中立 API。
- 不动 `EconomyUtil` 账户资金逻辑。

### 2.2 源码归属

当前状态：

- 根目录 `src/main/kotlin` 无业务源码残留。
- 不再使用 Gradle `sourceSets` include/exclude 临时路由旧源码。
- 各模块源码已物理移动到对应模块目录。

已知保留：

- 根 `build.gradle.kts` 和各模块构建脚本中的 `exclude(group = "org.apache.logging.log4j")` 是依赖排除，不是源码归属 hack。
- 根 `build.gradle.kts` 中的 jar include 是打包产物匹配，不是源码归属 hack。

## 4. 当前主要偏差与风险

### 4.1 `core -> image` 阶段性偏差

设计文档理想依赖是：

```text
core -> data/common
```

当前实际仍存在：

```text
core -> image
```

原因：

- 个人信息图、私人银行信息图等渲染编排仍在 core 流程中。
- 现在硬拆会扩大 main/action/presentation 层改动面。

处理策略：

- 2.0.0 收尾阶段允许保留。
- 后续等 image ViewModel 稳定后，再把渲染编排上移到 main 或独立 presentation 层。

### 4.2 真实 bot 链路缺证据

本地构建、测试和 runConsole 启动级冒烟已有证据，但仍缺：

- 群消息 Action 真实触发。
- 图片上传并发送到群。
- 私人银行群内查询或基础操作。
- 至少一个 game 入口响应。

处理策略：

- 这是当前最优先的结束前补验项。
- 需要可用 Overflow token 和测试群环境。

### 4.3 game 深层流程仍偏粗

game 已物理拆分，但玩法逻辑天然复杂，部分 usecase/service 仍较长。

处理策略：

- 暂不影响 2.0.0 主体结束。
- 等 data/core/main 稳定后，按玩法逐个拆短。

### 4.4 历史乱码文案仍多

影响范围：

- command 描述
- Action 回复
- manager/usecase 日志
- data proxy/repository 日志
- docs 结构说明

处理策略：

- 单独做文案专项。
- 不和架构迁移混在同一批。
- 涉及旧数据兼容的 mojibake 字符串必须保留注释说明。

## 5. 推荐整理路线

### 第一优先级：补齐 bot 侧最小冒烟

目标：

- 证明 2.0.0 模块化后的插件不只可构建，也能在真实 Mirai/Overflow/bot 链路中响应核心入口。

执行：

```powershell
.\gradlew.bat :economy-main:runConsole --console=plain
```

测试群内最小用例：

1. 触发主帮助图：`帮助`
2. 触发个人信息图：`个人信息` 或 `info`
3. 触发私人银行查询：`银行列表` 或 `银行信息`
4. 触发至少一个 game 入口：优先 `鱼塘等级`，权限允许时再测 `钓鱼`
5. Console 执行：`hye v`
6. Console 执行：`stop`

完成记录必须包含：

- 启动日志关键片段。
- 群内触发命令。
- 群内返回结果。
- 图片是否成功发送。
- 是否有异常堆栈。
- 停止卸载是否正常。

### 第二优先级：保持本地边界守护

每次架构批次结束后运行：

```powershell
.\gradlew.bat :economy-common:test :economy-data:test :economy-image:test --console=plain
.\gradlew.bat :economy-core:test :economy-game:test :economy-main:test --console=plain
.\gradlew.bat clean buildPlugin --console=plain
```

检查重点：

- common 不出现 Mirai/Hibernate/业务模块依赖。
- data 不反向依赖 core/game/main/image。
- image 不出现 Mirai 上传/发送依赖。
- core/game/main 不新增 `HibernateFactory` 直连。
- proxy module 不直接调用 `HibernateDataStore`。

### 第三优先级：core/data 小收口

目标：

- 不改变账户资金语义。
- 不替换 Mirai 发消息。
- 继续减少 core 对 data 内部细节的了解。

候选任务：

- 继续审计 core 中直接依赖 entity 行为的地方。
- 将明显属于 data 保存语义的 upsert/timestamp 编排下沉到 repository。
- 保持 `EconomyUtil` 账户逻辑不动。

禁止项：

- 不重写资金流。
- 不为了“干净”强行改旧数据兼容逻辑。
- 不把 Mirai 发送在本阶段硬抽象成平台 API。

### 第四优先级：image 输入模型细化

目标：

- 让 image 更像纯渲染模块。

候选任务：

- 继续整理 `image.model`。
- 按帮助图、个人信息图、私银图拆分更明确的渲染输入。
- 拆小 `EconomyImageRenderer` 中过长绘制片段。

禁止项：

- image 不引入 Mirai 上传 API。
- image 不引入 Mirai 消息发送类型。
- image 不依赖 core/game/main。

### 第五优先级：main Action 继续瘦身

目标：

- Action 只做入口层职责。

候选任务：

- 参数解析、权限入口、调用 usecase/service、发送结果保留。
- 复杂流程继续下沉到 core/game service。
- 事件订阅继续由 main 管理，但不要让 Action 兼任事件调度器。

### 第六优先级：文案乱码专项

目标：

- 修复用户可见文案和日志乱码。

顺序：

1. main command/action 文案
2. core manager/usecase 日志
3. data proxy/repository 日志
4. game 用户可见回复
5. docs 中过期结构说明

原则：

- 每批只修一个模块。
- 修完跑对应模块编译和 `buildPlugin`。
- 旧数据兼容必须保留 mojibake 时，写注释说明。

### 第七优先级：game 深收敛

目标：

- 在 2.0.0 主体稳定后，再按玩法拆小。

候选任务：

- 先拆 `RobUsecase` 的消息构造或结算片段。
- 再拆 `GamesUsecase` 的等待与结算片段。
- 抽奖派奖继续走 `LotteryPayoutService`，并逐步收口到 core 能力。
