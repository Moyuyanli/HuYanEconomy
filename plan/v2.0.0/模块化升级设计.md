# HuYanEconomy 2.0.0 模块化升级设计

更新时间：2026-07-03

本文用于记录 2.0.0 阶段的 Gradle 多模块化设计与迁移原则。2.0.0 的目标是让项目结构真正进入新阶段，但不在本轮强制完成平台无关化；主产物仍然是 Mirai Console 插件。

## 目标

- 2.0.0 以模块化重生为主，直接拥抱 Gradle 多模块结构。
- 主产物仍由 `economy-main` 打包为 Mirai 插件 jar。
- 当前阶段不以多入口兼容为交付目标，不新增 `economy-api` 或 `economy-proxy` 模块。
- 允许 `economy-core` / `economy-game` 在短期内保留 Mirai API 依赖，但新增代码应尽量避免扩大耦合面。

## 模块划分

正式模块固定为以下六个：

| 模块 | 职责 |
|------|------|
| `economy-main` | Mirai 插件入口、生命周期、配置加载、权限注册、Action、事件订阅、最终打包 |
| `economy-common` | 基础工具、通用 Result、时间/金额/文本等平台弱相关能力 |
| `economy-data` | 实体 V1/V2、DTO、Converter、Repository、EntityProxy、DataSourceStrategy、Cache、Migration |
| `economy-image` | 图片渲染核心、底图、字体、画布工具、组件化 Renderer |
| `economy-core` | 个人信息、签到、银行、转账、道具、称号、红包等核心经济流程 |
| `economy-game` | 钓鱼、抢劫、抽奖、农场等玩法模块 |

依赖方向：

```text
economy-main  -> economy-core, economy-game, economy-image, economy-data, economy-common
economy-game  -> economy-core, economy-data, economy-image, economy-common
economy-core  -> economy-data, economy-common
economy-data  -> economy-common
economy-image -> economy-common
economy-common -> 无项目内模块依赖
```

`economy-main` 可以依赖全部模块，但其他模块不得反向依赖 `economy-main`。

## Action 归属

Action 放在 `economy-main`。

Action 是 Mirai 指令入口适配层，不是业务层。它只负责：

- 接收 Mirai 事件与 HuYanAuthorize 注解扫描。
- 做指令参数解析、权限入口与简单校验。
- 调用 `economy-core` / `economy-game` 中的 usecase 或 manager。
- 将文本、图片、转发消息等结果发送回 Mirai。

业务计算、数据变更、定时任务逻辑、图片绘制逻辑不应继续沉淀在 Action 中。

## Data 与 Proxy 归属

当前项目中的 `proxy` 表示数据代理与多版本数据访问门面，应归入 `economy-data`，不单独拆成 `economy-proxy` 模块。

`economy-data` 负责：

- V1 / V2 实体与后续实体版本。
- DTO 与 Converter。
- Repository 与 Hibernate 访问封装。
- `EntityProxy`、`EntityProxyRegistry`、`DataSourceStrategy`、`DataVersion`。
- 数据迁移、回退链、缓存与一致性控制。

`proxy` 这个词只用于数据代理语义，不用于表示未来的平台入口代理，避免概念混淆。

## Image 模块策略

`economy-image` 是 2.0.0 后的重要基础能力。未来大部分用户可见功能会逐步转为图片输出，因此图片模块需要优先保持独立。

约束：

- 图片模块不直接依赖 Mirai 的上传、消息、群、用户类型。
- Renderer 接收 DTO / ViewModel / 简单值对象，输出 `BufferedImage` 或字节数据。
- Mirai 图片上传与发送逻辑留在 `economy-main`。
- 底图、字体、主题、布局组件、文本测量、表格/卡片绘制工具统一沉淀在 `economy-image`。

## Mirai 耦合策略

当前代码中 Mirai 类型不只存在于 Action 层，`usecase`、`manager`、`model`、`config`、`utils` 等目录也有直接引用。因此 2.0.0 不强制要求 `economy-core` / `economy-game` 完全脱离 Mirai API。

本轮接受的现实约束：

- `economy-core` 和 `economy-game` 可以暂时依赖 Mirai 的 `User`、`Group`、`MessageEvent` 等类型。
- 旧流程迁移时以可编译、可运行、模块职责清晰为优先。
- 不因为平台抽象而阻塞 2.0.0 模块化推进。

新增代码建议：

- 优先传递 `Long`、`String`、DTO、业务请求对象，而不是直接传递 Mirai 事件对象。
- 需要发送消息时，优先在 `economy-main` 完成发送。
- 需要用户身份时，优先沉淀为用户 id、群 id、昵称、权限状态等轻量数据。
- 不把新的 Mirai API 依赖引入 `economy-image` 和 `economy-data`。

## 迁移顺序

建议按以下顺序迁移，避免一次性重排导致不可控：

1. 拆出 `economy-common`
   - 移动平台弱相关工具、通用值对象、金额/时间/格式化工具。
   - 避免放入 Mirai、Hibernate、插件生命周期相关代码。

2. 拆出 `economy-data`
   - 移动实体、DTO、Converter、Repository、EntityProxy、DataSourceStrategy、缓存和迁移能力。
   - 保持 Hibernate 运行期依赖策略不变，避免重新引入 ClassLoader 冲突。

3. 拆出 `economy-image`
   - 移动图片绘制核心能力。
   - 将 Mirai 图片上传逻辑留在 `economy-main`。

4. 拆出 `economy-core`
   - 移动个人信息、签到、银行、转账、道具、称号、红包等核心经济流程。
   - 短期允许保留必要 Mirai 依赖。

5. 拆出 `economy-game`
   - 移动钓鱼、抢劫、抽奖、农场等玩法。
   - 玩法可依赖核心经济能力和图片能力。

6. 整理 `economy-main`
   - 保留插件入口、Action、配置加载、权限注册、事件订阅和最终打包。
   - 确认 KSP 仍能扫描 Action，`buildPlugin` 仍能生成 Mirai 插件产物。

## 未来多入口预留

本轮不创建 `economy-api` 模块，也不创建用于平台入口的 `economy-proxy` 模块。

如果未来要支持 Mirai 以外的入口，例如 HTTP、CLI、其他机器人平台，应另起一次设计，并考虑新增 adapter/API 层。届时可以逐步引入平台中立类型，例如：

```kotlin
data class EconomyActor(
    val platform: String,
    val userId: Long,
    val groupId: Long?,
    val displayName: String?
)

data class EconomyCommandContext(
    val actor: EconomyActor,
    val rawText: String,
    val args: List<String>
)

interface EconomyOutput {
    suspend fun text(message: String)
    suspend fun image(bytes: ByteArray)
}
```

这些类型只是未来方向，不属于当前 2.0.0 模块化拆分的强制实现内容。

## 验收标准

文档阶段不改源码，不需要运行构建。

真正执行模块拆分时，每完成一个迁移阶段运行：

```bash
./gradlew.bat buildPlugin --console=plain
```

阶段验收要求：

- `economy-main` 仍能生成 Mirai 插件产物。
- HuYanAuthorize KSP 权限扫描仍能发现 Action。
- Hibernate 实体、数据代理、数据版本策略仍能初始化。
- `economy-image` 不直接依赖 Mirai 消息发送或图片上传类型。
- `economy-data` 不承担平台入口适配职责。

## 默认决策

- 模块名使用 `economy-main`，不改成 `economy-mirai`。
- 当前不创建 `economy-api` 或 `economy-proxy` 模块。
- `proxy` 仅表示数据代理，归入 `economy-data`。
- 2.0.0 接受 core/game 中短期保留 Mirai API，未来通过文档约束逐步收敛。
## 待办清单

本清单记录 2.0.0 多模块拆分在当前进度之后仍需继续完成的工作。每一项都应以可构建、可启动、职责更清晰为验收标准，不为了形式上的纯净边界牺牲当前插件可用性。

### 1. economy-data 深拆与命名收敛

- 完成 `converter/**` 归属梳理。
  - V1/V2 实体转换器应迁入 `economy-data`。
  - 转换器不得依赖 `manager`、`usecase`、`action`、Mirai 消息类型。
  - 对仍引用业务对象的转换器先记录并解耦，再纳入 data。
- 完成 `repository/**` 归属梳理。
  - 纯 Hibernate 访问、查询、保存、删除逻辑进入 `economy-data`。
  - 带业务流程判断、权限判断、消息发送、定时任务调度的仓储代码不得直接进入 data，需要拆成 data repository + core service。
- 完成 `cache/**` 归属梳理。
  - `CacheConfig`、缓存接口、缓存一致性检查等平台无关部分进入 `economy-data`。
  - Redis 或其他缓存实现若引入外部依赖，应保持可选，不影响当前默认构建。
- 继续清理 `proxy/module/*` 的反向业务依赖。
  - `PrivateBankEntityProxy` 当前不应直接依赖私人银行业务仓储，应改为依赖 data repository 或更底层的 Hibernate 数据访问能力。
  - `UserEntityProxy` 中对背包代理的跨代理读取需要确认是否属于 data 组合查询，避免引入 core 行为。
  - 所有 proxy module 应只负责 V1/V2 读写、DTO 转换、迁移，不承载业务规则。
- 完成数据包名统一。
  - 当前 Kotlin 包名已倾向 `cn.chahuyun.economy.data.proxy`。
  - 后续应逐步把物理目录也迁移到 `economy-data/src/main/kotlin/cn/chahuyun/economy/data/...`，或至少保证源码路径与包名一致。
  - 迁移物理目录前必须确保无用户未合并改动被覆盖。
- 数据模块验收。
  - `economy-data` 不依赖 `economy-main`、`economy-core`、`economy-game`、`economy-image`。
  - `economy-data` 可单独 `compileKotlin`。
  - `EntityProxyRegistry` 初始化后仍能注册所有数据代理。
  - `DataSourceStrategyImpl` 仍能由 `economy-main` 注入配置并持久化 V2 模块版本。

### 2. economy-common 继续收口

- 梳理 `ShareUtils`。
  - 已将文本替换、概率命中、取整逻辑委托给 common。
  - 后续可拆为 `common` 纯工具 + `main` Mirai 工具，保留旧 API 作为兼容层。
- 拆分 `Constant.kt`。
  - `TOPIC` 这类纯常量可进入 common。
  - `CURRENCY_GOLD` 依赖 Mirai economy currency，不能直接进入 common，应保留在 main 或未来平台适配层。
- 清理并确认 common 常量。
  - `EconPerm`、`PropConstant`、`PropsKind`、`TitleCode`、`UserLocation` 已适合进入 common。
  - `EconPerm.GROUP` 和 `UserLocation.displayName` 中的历史乱码文案需要统一修复。
- 继续沉淀通用基础类型。
  - `EconomyResult` 可作为 core/data/game 之间返回业务结果的基础类型。
  - 后续新增流程优先返回 `EconomyResult<T>` 或明确 DTO，少用异常表示普通业务失败。
- common 模块验收。
  - 不依赖 Mirai、Hibernate、插件主类、业务 manager。
  - 不出现 `net.mamoe`、`HibernateFactory`、`HuYanEconomy` import。
  - 仅承载时间、金额、文本、日志门面、轻量常量、通用结果等底层能力。

### 3. economy-image 深拆

- 迁移图片渲染核心。
  - `EconomyImageRenderer`、画布工具、字体选择、底图选择、组件化 renderer 应继续向 `economy-image` 收口。
  - `ImageDrawXY` 这类图片坐标常量应归入 `economy-image`，不放 common。
- 明确图片模块输入输出。
  - 输入应为 DTO、ViewModel、简单值对象。
  - 输出应为 `BufferedImage`、`ByteArray` 或可测试的渲染结果。
  - 不直接上传 Mirai 图片，不直接发送 Mirai 消息。
- 清理反向依赖。
  - image 不依赖 `economy-main`。
  - image 不依赖业务 manager 或数据 repository。
  - 若需要业务数据，必须由 core/game/main 组装成渲染模型后传入。
- 图片模块验收。
  - `economy-image` 可单独 `compileKotlin`。
  - 主要图片渲染入口可由测试或预览工具直接调用。
  - main 只负责把 image 产物转换为 Mirai 消息发送。

### 4. economy-core 拆分

- 迁移核心经济流程。
  - 个人信息、签到、银行、转账、道具、称号、红包等业务流程逐步进入 `economy-core`。
  - core 可短期保留 Mirai API，但新增代码优先传 `Long`、`String`、DTO、业务请求对象。
- 拆分 manager/usecase 职责。
  - manager 更偏模块级生命周期和对外 API。
  - usecase 更偏单次业务流程编排。
  - 数据读写应通过 `economy-data` 的 proxy/repository，不直接散落 Hibernate 调用。
- 处理消息发送边界。
  - core 内尽量返回文本、图片请求、结果 DTO，而不是直接发送 Mirai 消息。
  - 短期无法抽离的 Mirai 发送逻辑要标记，后续迁移到 main adapter。
- core 模块验收。
  - 不依赖 `economy-main`。
  - 依赖方向为 `core -> data/common`，必要时短期依赖 Mirai compileOnly。
  - 核心流程迁移后原 Action 行为不变。

### 5. economy-game 拆分

- 迁移玩法模块。
  - 钓鱼、抢劫、抽奖、农场逐步进入 `economy-game`。
  - game 可依赖 core/data/image/common。
- 梳理游戏状态和数据访问。
  - 钓鱼、鱼塘、抽奖批次、农场库存等数据访问走 data proxy/repository。
  - 游戏奖励、扣款、称号发放等调用 core 能力，不直接复制经济逻辑。
- 清理 Mirai 扩散。
  - 玩法流程可短期接收 Mirai 事件。
  - 新增内部逻辑尽量转成用户 id、群 id、参数 DTO。
- game 模块验收。
  - 不依赖 `economy-main`。
  - 玩法 Action 仍在 main，调用 game usecase。
  - 钓鱼、抢劫、抽奖、农场主要命令完成冒烟测试。

### 6. economy-main 收口

- 保留 Mirai 插件入口职责。
  - 插件生命周期、配置加载、权限注册、Action、事件订阅、命令注册、最终打包保留在 main。
- Action 瘦身。
  - Action 只做参数解析、权限入口、调用 usecase、发送结果。
  - 不在 Action 中继续堆业务计算、数据库变更、图片绘制细节。
- 配置适配。
  - Mirai `AutoSavePluginConfig` 仍在 main。
  - 需要传给 data/core/game 的配置应转换为轻量配置对象或初始化参数。
- main 模块验收。
  - KSP 权限扫描仍能发现所有 Action。
  - `build/mirai/HuYanEconomy-2.0.0.mirai2.jar` 正常生成。
  - debug-sandbox 中插件可启动。

### 7. 编码与用户可见文案清理

- 系统性清理历史 mojibake 文案。
  - 权限组名称。
  - 用户位置显示名。
  - 日志内容。
  - 命令提示。
  - 红包、钓鱼、道具、称号等用户可见消息。
- 清理时注意不要改变业务语义。
  - 先修复缺失引号、断行、语法风险。
  - 再统一中文文案。
  - 最后考虑抽离到配置或消息模板。
- 编码验收。
  - 新增/改动文件统一 UTF-8。
  - 构建中不再出现因乱码导致的字符串截断。
  - 用户可见消息至少完成主要流程人工检查。

### 8. 物理目录迁移

- 当前阶段大量文件仍通过 Gradle sourceSet include/exclude 临时归属模块。
- 等模块依赖稳定后，逐步执行物理目录迁移。
  - `economy-common/src/main/kotlin/...`
  - `economy-data/src/main/kotlin/...`
  - `economy-image/src/main/kotlin/...`
  - `economy-core/src/main/kotlin/...`
  - `economy-game/src/main/kotlin/...`
  - `economy-main/src/main/kotlin/...`
- 迁移原则。
  - 每次只迁移一个模块或一个明确子域。
  - 迁移后立即运行 `./gradlew.bat clean buildPlugin --console=plain`。
  - 不在迁移同时做大规模业务重写。

### 9. 阶段验证计划

- 每完成一个拆分阶段运行：
  - `./gradlew.bat clean buildPlugin --console=plain`
  - 必要时补充 `./gradlew.bat :economy-data:compileKotlin --console=plain`
  - 必要时补充 `./gradlew.bat :economy-common:compileKotlin --console=plain`
- debug-sandbox 冒烟测试。
  - 插件启动。
  - 配置加载。
  - 权限注册。
  - Hibernate 初始化。
  - EntityProxyRegistry 初始化。
  - 签到、个人信息、银行、钓鱼、抢劫、抽奖、红包、农场至少各一条主命令。
  - 图片渲染功能能生成并发送。
- 回归关注点。
  - 插件 jar 是否仍排除 log4j 冲突依赖。
  - Hibernate/HikariCP 是否仍由运行时插件提供，不被错误打包。
  - KSP 生成代码是否仍与 Action 包扫描一致。
